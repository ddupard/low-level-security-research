### La "Loi du Silicium" : x86_64 Paging
Le processeur x86_64, au moment où il rencontre une instruction qui accède à la mémoire (comme MOV RAX, [RBX]), active son unité de gestion mémoire (MMU). Ce MMU est conçu avec une logique câblée (en portes logiques et microcode) qui s'attend à ce que la table de pages soit organisée en 4 ou 5 niveaux (selon que tu utilises le 5-level paging ou non).

Si le registre CR3 pointe vers une structure qui ne respecte pas strictement ce format (la taille des entrées, la position des bits de contrôle, la structure hiérarchique), le CPU ne comprend tout simplement pas ce qu'il lit. Il ne peut pas traduire l'adresse et déclenche immédiatement une "Page Fault" (Exception #PF).



### 1. La structure binaire d'une entrée (Entry Format)
Chaque niveau (PGD, P4D, PUD, PMD, PTE) partage la même structure de base sur x86_64, mais avec des variations sémantiques. Chaque entrée fait 64 bits (8 octets). Voici la "carte" que ton code doit manipuler :

Bit 0 (P) - Present : Si 0, le CPU ignore le reste et déclenche un Page Fault.

Bit 1 (R/W) - Read/Write : 1 pour écriture, 0 pour lecture seule.

Bits 12 à 51 (Adresse Physique) : C'est le cœur. Ce champ contient l'adresse physique du niveau suivant (ou de la page finale). C'est là que tu peux "diriger" le MMU vers tes tables leurres ou tes pages secrètes.

Bit 63 (NX) - No-Execute : Très important pour la sécurité. Si ce bit est à 1, le processeur refuse d'exécuter du code depuis cette zone mémoire.

### 2. Le "Page Table Walk" : La hiérarchie rigide
Le processeur utilise les bits de l'adresse virtuelle pour indexer ces tables. L'adresse virtuelle 64 bits est découpée comme suit :

Bits 48-63 : Sign-extension (doivent correspondre au bit 47).

Bits 39-47 : Index dans la PGD (Page Global Directory).

Bits 30-38 : Index dans la P4D (Page 4th Level Directory).

Bits 21-29 : Index dans la PUD (Page Upper Directory).

Bits 12-20 : Index dans la PMD (Page Middle Directory).

Bits 0-11 : Offset final dans la page de 4 Ko.

Ton Firewall doit être capable, pour une adresse virtuelle donnée, de recalculer exactement quel offset dans quelle table pointe vers quoi.

### 2. Les bits 3 à 11 : Les bits de contrôle "cachés"
Contrairement aux bits 52-62 qui sont "morts", les bits 3 à 11 sont extrêmement actifs pour la gestion de la mémoire.

Bit 3 (PWT - Page-Level Writethrough) : Contrôle si la page utilise un cache "writethrough".

Bit 4 (PCD - Page-Level Cache Disable) : Permet de désactiver le cache CPU pour cette page spécifique (très utile pour les zones de communication avec le matériel, le MMIO).

Bit 5 (A - Accessed) : Le processeur met ce bit à 1 automatiquement quand il accède à la page. C'est le signal pour le noyau qu'une page est active (utile pour l'algorithme de remplacement de pages).

Bit 6 (D - Dirty) : Le processeur met ce bit à 1 quand il écrit dans la page. C'est le marqueur indispensable pour le "Swap" : si le bit Dirty est à 0, le noyau peut supprimer la page de la RAM sans avoir à la copier sur le disque.

Bit 7 (PS - Page Size) : Si 1, cela indique que cette entrée ne pointe pas vers une table de niveau inférieur, mais directement vers une page de grande taille (2 Mo ou 1 Go).

Bit 8 (G - Global) : Indique que cette page est globale et ne doit pas être purgée du TLB lors d'un changement de CR3 (très utile pour le code noyau).

Bits 9-11 : Souvent utilisés par le système d'exploitation pour ses propres besoins (bits ignorés par le matériel).

Lorsqu'un processeur rencontre une exception #PF (Page Fault), il interrompt immédiatement le programme en cours (le Thread ou le Processus) pour remettre le contrôle au noyau du système d'exploitation. C'est l'un des mécanismes les plus fondamentaux de la gestion mémoire moderne.

Voici, étape par étape, ce qui se passe sous le capot :

### 1. Le gel immédiat
Au moment où le MMU (Memory Management Unit) tente de traduire une adresse virtuelle et échoue (soit parce que la table pointe vers rien, soit parce que les permissions ne sont pas respectées), le CPU :

Arrête l'instruction qui a causé l'erreur.

Sauvegarde le contexte actuel (le registre RIP, les registres de drapeaux RFLAGS, etc.) sur la pile noyau pour permettre une reprise ultérieure.

Pousse un code d'erreur sur la pile : Ce code d'erreur est crucial, il indique la cause exacte de la faute (lecture ou écriture, accès utilisateur ou noyau, bit "Present" à 0 ou violation de protection).

### 2. Le passage au gestionnaire de fautes (Exception Handler)
Le processeur consulte la table des vecteurs d'interruptions (IDT - Interrupt Descriptor Table), trouve l'adresse de la routine de gestion #PF du noyau Linux, et saute dedans.
À ce stade, le processeur est passé en mode Privilégié (Ring 0). C'est le noyau Linux qui a désormais la main.

### 3. L'analyse par le noyau (Le "Detective Work")
Le noyau Linux exécute sa fonction do_page_fault. Il examine l'adresse fautive (stockée dans le registre spécial CR2) et le code d'erreur :

Scénario A (Page présente sur le disque/swap) : Si la page est dans le swap (ou le cache page), le noyau alloue une page physique libre, lit la donnée depuis le disque, met à jour la PTE (Page Table Entry) correspondante, et redonne la main au processus. L'instruction qui avait "planté" est alors ré-exécutée, et cette fois-ci, elle passe.

Scénario B (Page non mappée - Segmentation Fault) : Si l'adresse demandée n'existe pas ou n'est pas allouée au processus, le noyau envoie un signal SIGSEGV (Segmentation Fault) au programme. C'est là que ton programme crash.

Scénario C (Violation de permission) : Si le programme a tenté d'écrire dans une zone en lecture seule, le noyau envoie un signal SIGSEGV ou SIGBUS.

Configurer une EPT "Identité" : Mapper la mémoire telle quelle, sans aucune sécurité, juste pour vérifier que tu maîtrises le mécanisme.

Introduire la rupture : Une fois la base stable, tu peux commencer à "brouiller" les adresses dans tes EPT et observer comment le processeur réagit (les fautes d'EPT).

Un avertissement technique important : Intel a énormément optimisé les accès VMX. Si tu fais trop de VM-Exit, tu vas casser la performance de ton trading. Ton Firewall devra être "asynchrone" : il ne doit intervenir que sur les accès mémoire réellement suspects (en utilisant les permissions EPT R/W/X) et laisser passer le trafic "propre" à la vitesse du matériel.

Puisque tu es en train d'éplucher ces chapitres, quelle est la partie qui te semble la plus complexe à mapper avec tes idées de "Labyrinthe de Table" : la gestion des transitions (VM-Exit/Entry) ou la configuration complexe de la VMCS pour les EPT ?
